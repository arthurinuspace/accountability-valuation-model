# 內部溝通簡報：技術部 (System Development) 專屬版

這份簡報設計旨在向技術團隊（工程師、PM、QA）說明新的當責評估模型。
技術人員通常**重視邏輯、討厭模糊、在意公平**，並擔心「做雜事（如文件、Technical Debt）」不被看見。
因此，本簡報將採用**「演算法解析」**的風格，強調規則的透明度與邏輯性。

---

## 簡報基本資訊
*   **目標聽眾**：技術部全體同仁 (Dev, PM, QA)。
*   **溝通目標**：
    1.  **消除疑慮**：證明這不是變相的 KPI 加壓，而是讓「隱形貢獻」被看見。
    2.  **建立共識**：讓大家認同「寫文件 (SOP)」和「救火 (Risk)」是有價值的。
    3.  **規則透明**：明確定義分數怎麼算。

---

## 簡報架構與內容 (Slide Outline)

### Part 1: The "Algorithm" of Value (價值的演算法)

#### Slide 1: 封面
*   **標題**：**Value = Impact(x) + Asset(+)**
*   **副標題**：解析全新的技術人才估值邏輯
*   **視覺風格**：程式碼風格 (Dark Mode IDE theme)，將公式寫成 pseudo-code。

#### Slide 2: 我們想解決的 Bug (現況痛點)
*   **內容**：
    1.  `Bug #1`: 努力寫扣，結果需求理解錯誤，重寫。（目標沒釐清）
    2.  `Bug #2`: 還債（Refactoring）和寫文件被認為是「沒產出」。
    3.  `Bug #3`: 幫別人救火，結果自己的進度延遲被扣分。
*   **定調**：「新的模型就是為了 `Fix` 這些 Bug 而設計的。」

#### Slide 3: The Kernel: 內外動力混合模型 (公式解析)
*   **呈現手法**：直接秀出公式圖，但用工程師的語言標註。
    *   **乘法區 (釐清 x 執行)** = `Core Process` (主要執行緒)
    *   **釐清 (0~1.2)** = `Spec Validation` (規格確認)
    *   **執行 (0~1.2)** = `Implementation` (實作交付)
    *   **規則**：`if (Spec == Wrong) return 0;` (強調目標對齊的絕對否決權)。

---

### Part 2: Feature Flags (各項維度詳解)

#### Slide 4: 關於「釐清」：不要盲目 Coding
*   **疑慮回應**：「我只是照 Spec 做，為什麼要我負責釐清？」
*   **解釋**：
    *   工程師的價值在於 `Feasibility Check` (可行性評估) 與 `Edge Case Detection`。
    *   **得分點**：在 `Kick-off`, `Planning` 會議中指出邏輯漏洞，避免後續重工。這就是高分。

#### Slide 5: 關於「SOP」：Code is NOT Documentation
*   **疑慮回應**：「我的 Code 寫得很乾淨，不需要文件。」
*   **解釋**：
    *   SOP = **可複用性 (Reusability) & 傳承 (Legacy)**。
    *   **得分點**：
        *   寫出 `README` 讓新人在 10 分鐘內架好環境。
        *   封裝一個 `Common Library` 讓其他專案共用。
        *   建立 `CI/CD` 流程自動化。
    *   這些都是**數位資產**，屬於加分項。

#### Slide 6: 關於「風險補償」：Firefighter Award
*   **疑慮回應**：「又要我救火，又要我準時交付？」
*   **解釋**：
    *   過去：救火耽誤進度 -> 績效變差。
    *   現在：救火有獨立的 `Risk Compensation` 分數。
    *   **得分點**：半夜起來修 `P0 Incident`，即使原專案 delay，這裡的分數會補回來，甚至更高。

---

### Part 3: FAQ (執行面的公平性與透明度)

#### Slide 7: 分數誰來打？ (透明度)
*   **機制**：
    *   不只是主管打印象分數。
    *   **舉證制 (Proof-based)**：
        *   釐清：拿出 Jira 上的討論紀錄/留言。
        *   SOP：拿出 Wiki 連結、Git Commit (Refactor/Docs)。
        *   超越：拿出效能監控數據 (Before/After)。
*   **Code Review 類比**：就像 PR 需要 Reviewer Approve 一樣，你的貢獻有憑有據。

#### Slide 8: 獎勵制度 (Incentive)
*   **連結**：
    *   **年終獎金**：連動總分 (Total Score)。
    *   **晉升 (Promotion)**：特別看重「增值盤 (加法區)」。
        *   *Senior 工程師不只要會寫 Code (乘法)，更要會建立架構與帶人 (加法)。*

#### Slide 9: Q&A 直球對決
*   **預擬問題**：
    *   Q: 「如果 PM 規格一直改，算誰的？」
    *   A: 這是「環境風險」。你有對齊（釐清分拿滿），你盡力改（風險補償拿滿）。不用擔心被扣執行分。

## 現場說明與解釋的方法

1.  **使用他們的語言**：
    *   多用 `Bug`, `Hotfix`, `Refactor`, `Spec`, `Technical Debt` 這些詞彙。讓他們覺得你是圈內人，不是來畫大餅的。

2.  **強調「保護機制」**：
    *   技術人員常覺得自己是下游受害者。強調這個公式中的 **「風險補償」**和 **「目標釐清」**其實是在 **保護工程師**。
    *   *「如果 PM 需求講不清楚，你拒絕開工，這是符合『目標釐清』精神的，不應該被視為配合度差。」* ——這句話會贏得很多掌聲。

3.  **舉證責任**：
    *   強調採用「Github 精神」：**Contributions need to be committed.**
    *   不用寫作文誇耀自己，貼連結就好。這會讓工程師覺得公平且輕鬆。

4.  **視覺呈現**：
    *   可以用 **Git Graph** 的分支圖來解釋：
        *   `Master` 分支是「基本執行」。
        *   `Feature` 分支是「創新/超越」。
        *   `Hotfix` 分支是「風險補償」。
    *   最後都要 Merge 回來算出總價值。
